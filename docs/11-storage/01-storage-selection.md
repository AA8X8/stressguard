## Выполнение задания «Системное мышление о выборе хранилища» для StressGuard

### 1. Определение сущностей, которые необходимо хранить

В проекте StressGuard выделим следующие ключевые сущности:

| Сущность | Атрибуты | Акторы (кто взаимодействует) | Характер взаимодействия |
|----------|----------|------------------------------|--------------------------|
| **Организация (Organization)** | orgId (PK), name, type (university/company) | HR-специалист (при регистрации организации), бэкенд | Справочные данные – редко меняются. |
| **Пользователь (User)** | userId (PK), fullName, email, phone, userType (student/employee/hr), orgId (FK), createdAt, isActive | Пользователь (при регистрации), HR-специалист (при управлении), бэкенд | Транзакционные данные – запись при регистрации, обновление профиля. |
| **Профиль пользователя (UserProfile)** | profileId (PK), userId (FK), userType, currentEntries, goalEntries, daysLeft, stressIndex, fitPermission, calendarPermission, moodEntered, fatigueEntered, sleepEntered, ratingGiven, fitDataAvailable, calendarDataAvailable, points, goalAchieved | Пользователь (через мобильное приложение), HR-специалист (через веб-дашборд), бэкенд-сервисы | Транзакционные данные – часто обновляются (каждый день: ввод данных, оценка, начисление баллов). Чтение при каждом открытии приложения. |
| **Дневная запись (DailyEntry)** | dailyEntryId (PK), profileId (FK), timestamp, moodValue, moodReason, fatigueValue, sleepQuality, sleepDuration, sleepLatency, fitnessData (JSON), calendarEvents (JSON) | Бэкенд-сервис сбора данных, ML-сервис, сервис аналитики | Транзакционные + аналитические – создаётся раз в день, потом используется для отчётов, трендов, агрегации HR. |
| **Рекомендация (Recommendation)** | recId (PK), text, category | Бэкенд (rule-движок), мобильное приложение (отображение) | Справочные данные – редко меняются (администратор добавляет/редактирует). Чтение частое. |
| **Уведомление о достижении цели (GoalAchievedEvent)** | eventId (PK), timestamp, profileId (FK), userType, currentEntries, goalEntries, achievedAt | Бэкенд (публикует), микросервис уведомлений (потребляет) | Поток событий – разовая запись, асинхронное чтение. |
| **Выдача бонуса (BonusIssuance)** | issuanceId (PK), profileId (FK), userType, goalEntries, achievedAt, periodStart, periodEnd, status (pending/issued/cancelled), issuedAt, issuedBy, comment | HR-специалист (через веб-дашборд), бэкенд (создаёт запись при достижении цели) | Транзакционные данные – создаётся автоматически при достижении цели (событие из RabbitMQ). HR обновляет статус на issued после ручной выдачи бонуса. |
| **Агрегированная статистика для HR (HR_dashboard_aggregates)** | org_id (FK), aggregated_at, avg_stress_index, active_users_count, goal_achieved_count, bonus_issued_count | HR-специалист (дашборд), сервис аналитики | Аналитические данные – пересчитываются периодически (раз в день или по запросу), только чтение. |

### 2. Выбор подходящих технологий хранения (алгоритм из лекции)

Пр  йдём по шагам для каждой сущности. Предположим, что объём данных:

- Пользователей: до 100 000 (в перспективе).
- Дневных записей: до 100 000 × 365 = 36,5 млн записей в год.
- Рекомендаций: несколько сотен.
- Событий достижения цели: до 100 000 в год.
- Агрегированная статистика: несколько тысяч строк (по организациям/отделам).

### Итоговый выбор типа хранилища

| Сущность | Тип хранилища | Обоснование |
|----------|---------------|-------------|
| **UserProfile** | **Реляционная СУБД** (PostgreSQL) | OLTP, сильная консистентность, жёсткая схема, индексы по userId, поддержка транзакций (хотя не критичны). |
| **DailyEntry** | **Реляционная СУБД + TimescaleDB** (расширение) | Основные данные – в PostgreSQL, временные ряды (метрики стресса, пульса) – в TimescaleDB для эффективной агрегации по времени. Гибкие поля – JSONB. |
| **Recommendation** | **Реляционная СУБД** (PostgreSQL) | Справочные данные, мало объёма, простота. |
| **GoalAchievedEvent** | **Брокер сообщений (RabbitMQ) + возможно реляционная БД для аудита** | Событие не требует долгого хранения. RabbitMQ – очередь. Для аудита можно сохранять в PostgreSQL. |
| **HR-агрегаты** | **Виртуальная витрина на основе PostgreSQL (материализованные представления) или отдельная колончатая БД (ClickHouse) при большом объёме** | Для начала достаточно материализованных представлений в PostgreSQL. При росте данных – ClickHouse. |

### 3. Дополнительные пояснения

- **Почему не Document DB для DailyEntry?**  
  DailyEntry имеет жёсткую структуру (mood, fatigue, sleep) плюс опциональные JSON-поля. PostgreSQL с JSONB отлично справляется с этим, а также поддерживает эффективные индексы и временные агрегаты через TimescaleDB.

- **Почему TimescaleDB?**  
  Позволяет хранить временные ряды (пульс, стресс-индекс по дням) с автоматическим партиционированием по времени, что ускоряет запросы «за последнюю неделю/месяц».

- **Асинхронные события:** GoalAchievedEvent не требует персистентного хранения в БД – достаточно очереди RabbitMQ. При необходимости можно сохранять в отдельную таблицу для аудита (тогда та же реляционная БД).

- **HR-агрегаты:** Для малого объёма подойдёт PostgreSQL с материализованными представлениями, обновляемыми по расписанию (например, раз в час). При масштабировании – ClickHouse (колончатая БД для аналитики).

### 4. Выбор технологии хранения данных

| Шаг | UserProfile | DailyEntry | Recommendation | GoalAchievedEvent | HR-агрегаты |
|-----|-------------|------------|----------------|-------------------|-------------|
| Сколько данных? | 100k записей (~10 МБ) | 36 млн записей (~10 ГБ/год) | сотни записей | до 100k в год | тысячи записей |
| Паттерн доступа | OLTP (update/read по ключу) | OLTP (write раз в день) + OLAP | OLTP (частое чтение) | Event streaming | OLAP (агрегация) |
| Консистентность | Strong | Strong | Strong | Eventual | Eventual |
| Доступность | 99.9% | 99.9% | 99.9% | 99.9% | 99.9% |
| Мутабельность схемы | Жёсткая | Гибкая (JSONB) | Жёсткая | Жёсткая | Денормализованная |
| Транзакции | не требуются | не требуются | не требуются | не требуются | не требуются |
| Запросы | по userId | по userId+дата, агрегация | по id, категории | по userId, дате | группировка, средние |
| Стоимость | дешёво | дешёво | дешёво | дешёво | дешёво |
| **Итоговое решение** | PostgreSQL | PostgreSQL + TimescaleDB | PostgreSQL | RabbitMQ (+ опционально PostgreSQL) | PostgreSQL (materialized views) / ClickHouse |

Таким образом, для StressGuard оптимально использовать **гибридный подход (polyglot persistence)**:

- Операционные данные – PostgreSQL.
- Временные ряды – TimescaleDB (расширение PostgreSQL).
- Асинхронные события – RabbitMQ.
- Аналитика – материализованные представления в PostgreSQL с возможностью миграции на ClickHouse при росте.

Это соответствует требованиям задания, учитывает объёмы, паттерны доступа и консистентность.
