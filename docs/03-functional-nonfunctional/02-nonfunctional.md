**Скоуп:** MVP (версия 1.0) – мобильное приложение для конечных пользователей, веб-дашборд для HR/деканата, бэкенд с ML-модулем и интеграциями.

---

## Категория: Безопасность (Security)

| ID | Название | Описание | Приоритет | Обоснование |
|----|----------|----------|-----------|-------------|
| NFR-SEC-01 | Шифрование данных при передаче | Все соединения между клиентскими приложениями (мобильное приложение, веб-дашборд) и сервером должны быть защищены протоколом TLS версии не ниже 1.2 (рекомендуется 1.3). | Must | Защита персональных данных и данных о здоровье от перехвата. |
| NFR-SEC-02 | Хеширование паролей | Пароли пользователей должны храниться в базе данных в виде хеша с использованием bcrypt (cost factor ≥12) или аналогичного алгоритма. | Must | Предотвращение компрометации учётных записей при утечке базы данных. |
| NFR-SEC-03 | Разграничение доступа к данным | Доступ к данным должен быть строго разграничен: пользователь видит только свои личные данные; HR/деканат видит только агрегированную анонимную статистику по своей организации; администратор видит данные всех организаций, но не может просматривать индивидуальные записи пользователей. | Must | Соблюдение конфиденциальности и требований законодательства. |
| NFR-SEC-04 | Шифрование чувствительных данных в БД | Данные о состоянии здоровья (пульс, HRV, сон, субъективные оценки) должны храниться в зашифрованном виде (например, AES-256) на уровне базы данных или с использованием механизмов шифрования на стороне приложения. | Should | Дополнительный уровень защиты на случай несанкционированного доступа к серверу. |
| NFR-SEC-05 | Блокировка при множественных неудачных попытках входа | При пяти неудачных попытках входа подряд учётная запись должна блокироваться на 15 минут. | Should | Защита от подбора паролей (brute-force). |

## Категория: Производительность (Performance)

| ID | Название | Описание | Приоритет | Обоснование |
|----|----------|----------|-----------|-------------|
| NFR-PERF-01 | Время загрузки главного экрана | Главный экран приложения (дашборд с индикатором стресса, графиком и рекомендациями) должен загружаться не более чем за 2 секунды для 95% пользователей при типовой нагрузке. | Must | Обеспечение комфортного пользовательского опыта; медленная загрузка снижает вовлечённость. |
| NFR-PERF-02 | Время отклика API | API-запросы (кроме операций, требующих ML-расчётов) должны обрабатываться не более 500 мс при нагрузке до 1000 запросов в минуту. | Should | Обеспечение отзывчивости интерфейса и фоновых синхронизаций. |
| NFR-PERF-03 | Время расчёта стресс-индекса | ML-модель должна вычислять стресс-индекс для пользователя не более 5 минут после получения всех необходимых данных за день (фоновый процесс). | Could | Пользователь не ожидает мгновенного результата, но важно, чтобы расчёт завершался до следующего входа. |
| NFR-PERF-04 | Использование ресурсов сервера | Средняя загрузка CPU на сервере приложений не должна превышать 70% при пиковой нагрузке (200 одновременных пользователей). | Should | Обеспечение стабильности и запаса для масштабирования. |

## Категория: Надёжность (Reliability)

| ID | Название | Описание | Приоритет | Обоснование |
|----|----------|----------|-----------|-------------|
| NFR-REL-01 | Доступность системы (Uptime) | Доступность системы для пользователей должна составлять не менее 99.5% в месяц (допустимый простой – около 3.5 часов в месяц). | Must | Критично для доверия клиентов (организаций) и непрерывности сбора данных. |
| NFR-REL-02 | Резервное копирование базы данных | Полное резервное копирование базы данных должно выполняться ежедневно в ночное время, с хранением копий не менее 30 дней. | Must | Защита от потери данных при сбоях; возможность восстановления. |
| NFR-REL-03 | Обработка ошибок интеграций | При сбое в работе внешних API (Google Fit, календари) система должна продолжать работу, используя последние доступные данные, и автоматически повторять попытку синхронизации сэкспоненциальной задержкой. | Should | Обеспечение устойчивости к временным проблемам внешних сервисов. |
| NFR-REL-04 | Восстановление после сбоя | Время восстановления системы после критического сбоя (RTO) не должно превышать 2 часов. | Should | Минимизация простоев для пользователей. |

## Объяснение по непокрытым категориям

В данном разделе выбрано **6 категорий** нефункциональных требований: Безопасность, Производительность, Надёжность, Масштабируемость, Совместимость и Нормативно-правовые. Эти категории являются наиболее критичными для успеха MVP и соответствия законодательству.

### 1. Емкость (Capacity)

- **Почему не критична для MVP:** На начальном этапе объём данных относительно невелик. Ожидаемое количество пользователей в пилоте – до 1000, что генерирует не более нескольких гигабайт метрик в месяц. Хранение таких объёмов не требует специального проектирования; стандартные базы данных (PostgreSQL) справятся без проблем.
- **Риски:** Если проект быстро вырастет, может потребоваться миграция на специализированные хранилища (например, TimescaleDB для временных рядов) и оптимизация хранения.
- **План:** В следующих версиях (после MVP) будут введены требования к ёмкости: максимальный объём данных, политика архивирования, ротация старых записей.

### 2. Наблюдаемость (Observability)

- **Почему не критична для MVP:** Для пилотного проекта достаточно базового логирования ошибок и мониторинга на уровне инфраструктуры (доступность сервера, загрузка CPU). Развёртывание полноценной системы логирования и алертинга может быть отложено до момента, когда появятся реальные пользователи.
- **Риски:** Сложность диагностики проблем в случае сбоев, особенно связанных с интеграциями или ML-моделью.
- **План:** На этапе разработки будут внедрены простые логи (в файлы или stdout), а после запуска пилота планируется настроить централизованное логирование (например, ELK или Sentry) и алерты по критичным ошибкам.

### 3. Удобство использования (Usability)

- **Почему не полностью покрыта:** Хотя удобство критически важно для вовлечения пользователей, оно в значительной степени обеспечивается на этапе проектирования интерфейса (UX/UI) и тестирования прототипов. В рамках НФТ мы можем указать лишь общие ориентиры, но детальные требования к юзабилити обычно разрабатываются в процессе дизайна и проверяются с реальными пользователями. В MVP мы опираемся на стандартные гайдлайны платформ (iOS Human Interface, Material Design) и интуитивно понятные сценарии, описанные в пользовательских историях.
- **Риски:** Низкая вовлечённость из-за неудобного интерфейса.
- **План:** После первого пилота будет проведено юзабилити-тестирование, на основе которого будут сформулированы конкретные НФТ по удобству (например, время выполнения ключевых сценариев, количество кликов и т.д.).

### 4. Поддерживаемость (Maintainability)

- **Почему не критична для MVP:** На этапе MVP команда разработки мала (1–3 человека), и кодовая база относительно небольшая. Требования к документированию и покрытию тестами важны, но их строгое соблюдение может замедлить разработку. Однако некоторые элементы (например, документирование API) уже включены в планы разработки.
- **Риски:** Накопление технического долга, затруднение дальнейшего развития.
- **План:** В требования к разработке будет включено обязательное документирование API (OpenAPI) и модульное тестирование ключевых модулей (бизнес-логика, ML-расчёты). После MVP планируется ввести требования к покрытию кода (например, 60% для критичных модулей) и стандарты кодирования.
